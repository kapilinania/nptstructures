## 🗺️ Where do I go from here?

If you've noticed a bug or have a feature request, **make an issue**! It's generally best if you get confirmation of your bug or approval for your feature request this way before starting to code. This saves everyone's time!

## 🐛 Bug Reports

A bug is a _demonstrable problem_ that is caused by the code in the repository. Good bug reports are extremely helpful, so thanks!

<details>
  <summary><b>Guidelines for submitting an excellent Bug Report:</b></summary>
  <br>

  > **Pro Tip:** Before submitting, please check if the issue has already been reported.

  1. 🔍 **Use the GitHub issue search** &mdash; check if the issue has already been reported.
  2. 🔄 **Check if the issue has been fixed** &mdash; try to reproduce it using the latest `main` or development branch.
  3. 🔬 **Isolate the problem** &mdash; ideally create a reduced test case and provide a clear, step-by-step procedure to reproduce it.

  **What to include:**
  - OS version and Browser (e.g., Windows 11, Chrome v114)
  - Clear steps to reproduce
  - Expected behavior vs Actual behavior
  - Screenshots or screen recordings (if visual)
</details>

## 🚀 Pull Requests

Good pull requests—patches, improvements, new features—are a fantastic help. They should remain focused in scope and avoid containing unrelated commits.

> **⚠️ WARNING:**  
> **Please ask first** before embarking on any significant pull request (e.g. implementing features, refactoring code, porting to a different language), otherwise you risk spending a lot of time working on something that the project's developers might not want to merge.

<details>
  <summary><b>🛠️ Step-by-Step Pull Request Process</b></summary>
  <br>

  ### 1. Setup Your Environment
  - Fork the repo and create your branch from `main`.
  - Clone your fork locally: `git clone https://github.com/YOUR_USERNAME/nptstructures.git`

  ### 2. Make Your Changes
  - Write clean, understandable code.
  - If you've added code that should be tested, **add tests**.
  - If you've changed APIs or UI components, **update the documentation**.

  ### 3. Verify
  - Ensure the test suite passes (if any).
  - Make sure your code lints and adheres to our Style Guide.

  ### 4. Submit
  - Push to your fork and submit that pull request!
  - Provide a clear and comprehensive description of the changes in the PR.
</details>

## 🎨 Code Style Guidelines

To keep our codebase clean, maintainable, and readable, please adhere to the following standards:

| Category | Guidelines |
| :--- | :--- |
| **Indentation** | Use consistent indentation (2 or 4 spaces, matching the existing file). |
| **HTML** | Ensure semantic HTML5 tags are used for better accessibility and SEO. |
| **CSS** | Ensure CSS is fully responsive. Always test your changes on mobile views! |
| **JavaScript** | Add concise comments for complex logic. Avoid massive functions. |
